01 — What You're Building
Core Idea
An event-driven microservices system that simulates the full lifecycle of a payment transaction — from initiation through risk screening, account debit, ledger posting, and notification — using patterns drawn from your Mastercard and Axis Bank production systems. Every inter-service call goes through Kafka. No direct REST between internal services.
Command Side

Write Model (CQRS)

All mutations go through commands handled by Axon aggregates. The aggregate enforces business invariants, emits domain events, and never reads its own query model. You'll implement this in payment-service using @CommandHandler and @EventSourcingHandler annotations.

Query Side

Read Model (Projections)

Event handlers maintain denormalized read tables in PostgreSQL. The query service reads these — zero joins, sub-millisecond reads, completely decoupled from write throughput. You'll implement this in query-service as a separate Spring Boot app with its own database.

Event Bus

Kafka as the Backbone

All inter-service communication flows through Kafka topics. Services are consumers and producers. No direct REST calls between internal services after the gateway. You already did this at Mastercard — this is the same pattern, your own implementation.

Failure Handling

Sagas + Compensating Transactions

Distributed transactions are managed by Axon Sagas. If any step fails — risk rejection, insufficient funds, timeout — compensating commands undo the in-progress steps. This is the same pattern as your Axis Bank builder loan lifecycle stages.

Reliability

Outbox Pattern

The Payment Service writes to its own DB and an outbox table in a single ACID transaction. A relay process polls and publishes to Kafka — eliminating dual-write failure risk. You need to implement: outbox_events table, a @Scheduled poller, and Kafka ACK tracking.

Idempotency

Exactly-Once Guarantees

Kafka producers use idempotent mode + transactional IDs. Consumers track processed event IDs in PostgreSQL. Duplicate events are detected and silently discarded. Implement a processed_events table in every consumer service.

02 — Build Order
What To Build First
Don't build everything in parallel. Follow this exact sequence — each step depends on the previous one being done. The common-service and Docker Compose go first because everything else depends on them.
Rule: Don't skip ahead. Each step assumes the previous one is working and tested locally. Push to GitHub after each step. Your repo should always have a working main branch.
01
Docker Compose + Infrastructure
Set up the local infrastructure: Kafka (with Zookeeper), Axon Server, and three PostgreSQL instances (payment_db:5432, account_db:5433, query_db:5434). Verify all containers start with docker compose up -d and you can connect to each DB. Create a docker-compose.yml at the project root.

common-service — Shared DTOs, Commands, Events
This is a non-runnable Maven module. Contains all shared classes: commands (CreatePaymentCommand, ConfirmPaymentCommand, CancelPaymentCommand), events (PaymentInitiatedEvent, AccountDebitedEvent, etc.), DTOs (Sender, Recipient, Address), and the Transaction entity. Every other module depends on this. You already have this partially built.
DEPENDS ON: Docker Compose running
03
api-gateway — REST Entry Point
Your current gateway handles REST + validation + idempotency cache. Next step: Wire it to send CreatePaymentCommand to Axon's CommandGateway instead of just returning an empty response. The gateway should return 202 Accepted with a payment ID. Add the Repeat-Flag / idempotency-key logic to actually return cached responses.
DEPENDS ON: common-service, Axon Server running
04
payment-service — Axon Aggregate + CQRS + Outbox
The core write model. Implement PaymentAggregate with @CommandHandler for Create/Confirm/Cancel commands and @EventSourcingHandler for state mutation. Then implement the Outbox pattern: outbox_events table, OutboxEntity, OutboxRepository, a @Scheduled poller that reads unpublished rows and sends them to Kafka, then marks them published after ACK. This is the hardest service. Get it right.
DEPENDS ON: common-service, Axon Server, Kafka, PostgreSQL (payment_db)
05
risk-service — Fraud Rule Engine
Kafka consumer that listens to payment.events topic. Runs configurable fraud rules against each payment. Publishes RiskApprovedEvent or RiskRejectedEvent. Simplest consumer service — good for validating your Kafka pipeline works end-to-end.
DEPENDS ON: common-service, Kafka, payment-service publishing events
06
account-service — Balance Management
Kafka consumer that debits/credits accounts. Needs its own PostgreSQL (account_db). Must implement optimistic locking on balance updates and idempotent event processing. Publishes AccountDebitedEvent or AccountDebitFailedEvent.
DEPENDS ON: common-service, Kafka, PostgreSQL (account_db)
07
payment-saga-service — Saga Orchestrator
Axon Saga that listens for events from risk-service and account-service. If both succeed → sends ConfirmPaymentCommand. If either fails → sends CancelPaymentCommand. This is where compensating transactions happen. Use @SagaEventHandler and @StartSaga / @EndSaga annotations.
DEPENDS ON: common-service, Axon Server, risk-service + account-service publishing events
08
query-service — Read Model + Projections
Kafka consumer that maintains denormalized projection tables (payment_summary, account_balance_view, payment_audit_log) in its own PostgreSQL (query_db). Exposes REST endpoints for status polling. This is the CQRS read side.
DEPENDS ON: common-service, Kafka, PostgreSQL (query_db), all events flowing
09
notification-service — Webhooks + DLT
Final consumer. Listens to all terminal events. Fires webhook stubs (just log for now). Implements retry with exponential backoff and Dead Letter Topic routing. Lowest priority — build last. Separate consumer group so it never blocks the payment flow.
DEPENDS ON: common-service, Kafka, all events flowing
03 — Architecture
System Design
Single API Gateway entry point. Commands flow through the Payment Service (CQRS write model). Events propagate via Kafka to independent consumer services. Query reads from projected PostgreSQL views. Each service owns its database — no shared schemas.

Outbox Pattern Implementation
// 1. OutboxEvent entity @Entity @Table(name = "outbox_events") public class OutboxEvent { @Id @GeneratedValue private Long id; private String aggregateId; private String eventType; // e.g. "PaymentInitiatedEvent" @Column(columnDefinition = "TEXT") private String payload; // JSON serialized event private boolean published; // false on creation private Instant createdAt; private Instant publishedAt; } // 2. OutboxEventListener — writes outbox row when aggregate emits event @Component public class OutboxEventListener { @EventHandler public void on(PaymentInitiatedEvent event) { OutboxEvent outbox = new OutboxEvent(); outbox.setAggregateId(event.getPaymentId()); outbox.setEventType("PaymentInitiatedEvent"); outbox.setPayload(objectMapper.writeValueAsString(event)); outbox.setPublished(false); outbox.setCreatedAt(Instant.now()); outboxRepository.save(outbox); // This runs in SAME transaction as Axon's event store write } } // 3. OutboxRelay — scheduled poller @Component public class OutboxRelay { @Scheduled(fixedDelay = 200) // 200ms poll interval public void publishPendingEvents() { List<OutboxEvent> pending = outboxRepository .findByPublishedFalseOrderByCreatedAtAsc(); for (OutboxEvent event : pending) { kafkaTemplate.send("payment.events", event.getPayloadAsJSON()) .whenComplete((result, ex) -> { if (ex == null) { event.setPublished(true); event.setPublishedAt(Instant.now()); outboxRepository.save(event); } }); } } }

Risk Service
FRAUD DETECTION · RULE ENGINE · ASYNC SCREENING · PORT :8082

Kafka consumer that screens every payment against configurable fraud rules before account debit. A rejection triggers full Saga rollback. This is your simplest consumer — build it first to validate the Kafka pipeline end-to-end.

Package Structure
com.suresh.paymentsimulator.risk ├── consumer/ │ └── PaymentEventConsumer // @KafkaListener on payment.events ├── rules/ │ ├── RiskRule // interface — evaluate(PaymentInitiatedEvent) │ ├── VelocityRule // > N txns per minute from same sender │ ├── AmountThresholdRule // single payment above configurable limit │ ├── BlocklistRule // sender or receiver on block list │ └── CurrencyMismatchRule // sender and receiver currency mismatch ├── service/ │ └── RiskEvaluationService // runs all rules, returns approve/reject ├── producer/ │ └── RiskEventProducer // publishes to risk.events topic ├── config/ │ ├── KafkaConsumerConfig // consumer group: risk-service │ ├── KafkaProducerConfig │ └── RiskRulesConfig // configurable thresholds from application.yml └── RiskServiceApplication
How It Works
@KafkaListener(topics = "payment.events", groupId = "risk-service") public void consume(PaymentInitiatedEvent event) { // 1. Deserialize event from Kafka message // 2. Run all RiskRule implementations against it // 3. If ALL pass → publish RiskApprovedEvent // 4. If ANY fail → publish RiskRejectedEvent with reason + rule name RiskResult result = riskEvaluationService.evaluate(event); if (result.isApproved()) { producer.send("risk.events", new RiskApprovedEvent( event.getPaymentId(), Instant.now())); } else { producer.send("risk.events", new RiskRejectedEvent( event.getPaymentId(), result.getReason(), result.getRuleViolated(), Instant.now())); } }
Fraud Rules — What To Implement
  • VelocityRule: Track sender transactions in a time window. If > risk.velocity.max-per-minute (configurable, default 5) → reject. Use an in-memory sliding window or a simple DB counter.
  • AmountThresholdRule: If amount > risk.amount.threshold (configurable, default 500000) → reject. Simple comparison.
  • BlocklistRule: Maintain a Set of blocked account numbers (load from config or a simple table). If sender or receiver is in the set → reject.
  • CurrencyMismatchRule: If the payment currency doesn't match the sender's expected currency → reject. For now, just validate against a whitelist of supported currencies.
Kafka Topics
  • Consumes: payment.events (group: risk-service)
  • Produces: risk.events

Account Service
BALANCE MANAGEMENT · DEBIT / CREDIT · IDEMPOTENT · PORT :8083

Consumes PaymentInitiatedEvent from Kafka and attempts to debit the sender account. Publishes the result as a new event. Critical: Must handle idempotent processing (Kafka redelivery) and optimistic locking (concurrent debits).

Package Structure
com.suresh.paymentsimulator.account ├── consumer/ │ └── PaymentEventConsumer // @KafkaListener on payment.events ├── entity/ │ ├── Account // @Entity — account_id, balance, currency, version │ └── ProcessedEvent // @Entity — event_id, processed_at ├── repository/ │ ├── AccountRepository // JpaRepository │ └── ProcessedEventRepository // JpaRepository — for idempotency ├── service/ │ └── AccountService // debit(), credit(), checkBalance() ├── producer/ │ └── AccountEventProducer // publishes to account.events └── config/ ├── KafkaConsumerConfig └── KafkaProducerConfig
How The Debit Flow Works
@KafkaListener(topics = "payment.events", groupId = "account-service") @Transactional public void consume(PaymentInitiatedEvent event) { // 1. IDEMPOTENCY CHECK — has this event already been processed? if (processedEventRepo.existsById(event.getPaymentId())) { log.info("Duplicate event {}, skipping", event.getPaymentId()); return; } // 2. FIND SENDER ACCOUNT Account sender = accountRepo.findByAccountNumber( event.getSenderAccountNumber()) .orElseThrow(() -> new AccountNotFoundException(...)); // 3. CHECK BALANCE if (sender.getBalance().compareTo(event.getAmount()) < 0) { // Insufficient funds → publish failure event producer.send("account.events", new AccountDebitFailedEvent( event.getPaymentId(), event.getSenderAccountNumber(), "INSUFFICIENT_FUNDS", Instant.now() )); processedEventRepo.save(new ProcessedEvent(event.getPaymentId())); return; } // 4. DEBIT — with optimistic lock (@Version) sender.setBalance(sender.getBalance().subtract(event.getAmount())); accountRepo.save(sender); // @Version prevents concurrent overwrites // 5. MARK AS PROCESSED + PUBLISH SUCCESS processedEventRepo.save(new ProcessedEvent(event.getPaymentId())); producer.send("account.events", new AccountDebitedEvent( event.getPaymentId(), event.getSenderAccountNumber(), event.getAmount(), Instant.now() )); }

Payment Saga Service
SAGA ORCHESTRATOR · COMPENSATION · DISTRIBUTED TRANSACTIONS · PORT :8086

Axon Saga that coordinates the multi-step payment flow. Listens for events from risk-service and account-service. If both succeed → sends ConfirmPaymentCommand. If either fails → sends CancelPaymentCommand to trigger compensation.

Package Structure
com.suresh.paymentsimulator.saga ├── PaymentSaga // THE Saga class — all logic here ├── config/ │ └── AxonConfig // Axon Server connection └── SagaServiceApplication
PaymentSaga — The Complete Class
public class PaymentSaga { // Transient — injected by Axon, not serialized @Autowired private transient CommandGateway commandGateway; // Saga state — serialized between events private String paymentId; private boolean riskApproved = false; private boolean accountDebited = false; // ── STARTED by PaymentInitiatedEvent ── @StartSaga @SagaEventHandler(associationProperty = "paymentId") public void on(PaymentInitiatedEvent event) { this.paymentId = event.getPaymentId(); // Just record — wait for risk + account results } // ── Risk approved ── @SagaEventHandler(associationProperty = "paymentId") public void on(RiskApprovedEvent event) { this.riskApproved = true; tryConfirm(); } // ── Account debited ── @SagaEventHandler(associationProperty = "paymentId") public void on(AccountDebitedEvent event) { this.accountDebited = true; tryConfirm(); } // ── BOTH succeeded → confirm payment ── private void tryConfirm() { if (riskApproved && accountDebited) { commandGateway.send(new ConfirmPaymentCommand(paymentId)); SagaLifecycle.end(); } } // ── Risk rejected → cancel payment ── @SagaEventHandler(associationProperty = "paymentId") public void on(RiskRejectedEvent event) { commandGateway.send(new CancelPaymentCommand( paymentId, "RISK_REJECTED: " + event.getReason())); SagaLifecycle.end(); } // ── Account debit failed → cancel payment ── @SagaEventHandler(associationProperty = "paymentId") public void on(AccountDebitFailedEvent event) { commandGateway.send(new CancelPaymentCommand( paymentId, "INSUFFICIENT_FUNDS")); SagaLifecycle.end(); } }
Key detail: associationProperty = "paymentId" is how Axon routes events to the correct Saga instance. Every event class MUST have a getPaymentId() method. Axon uses this to find the Saga instance that's tracking that specific payment.

How Projections Work
@KafkaListener(topics = {"payment.events", "risk.events", "account.events"}, groupId = "query-service") public void consume(String message) { // Deserialize → determine event type → update projections // PaymentInitiatedEvent → INSERT into payment_summary (status=INITIATED) // RiskApprovedEvent → UPDATE payment_summary SET status=RISK_APPROVED // RiskRejectedEvent → UPDATE payment_summary SET status=RISK_REJECTED // AccountDebitedEvent → UPDATE payment_summary SET status=DEBITED // → UPDATE account_balance_view (reduce balance) // AccountDebitFailedEvent → UPDATE payment_summary SET status=DEBIT_FAILED // PaymentConfirmedEvent → UPDATE payment_summary SET status=CONFIRMED // PaymentCancelledEvent → UPDATE payment_summary SET status=CANCELLED // ALWAYS: INSERT into payment_audit_log for every event }

06 — Transaction Flows
Happy Path vs Failure Path
Both paths must be fully implemented and testable. The failure path is not an afterthought — it demonstrates the complete Saga compensation cycle.
Happy Path — Successful Payment
  • 1Client sends POST /payments/initiate with idempotency keyREST
  • 2Gateway validates, issues CreatePaymentCommand to AxonCMD
  • 3PaymentAggregate emits PaymentInitiatedEvent to event store + outboxEVENT
  • 4Outbox relay publishes event to payment.events Kafka topicKAFKA
  • 5Risk Service consumes, runs rules, emits RiskApprovedEventRISK ✓
  • 6Account Service debits sender, emits AccountDebitedEventDEBIT ✓
  • 7Saga receives both events, sends ConfirmPaymentCommandSAGA
  • 8Aggregate emits PaymentConfirmedEvent; Query Service updates projectionDONE
  • 9Notification Service fires webhook. Client polls GET → CONFIRMEDNOTIF
Failure Path — Insufficient Funds
  • 1Steps 1–4 identical to happy path — payment initiated, published to KafkaINIT
  • 2Risk Service approves → RiskApprovedEventRISK ✓
  • 3Account Service checks balance — insufficient funds detectedCHECK
  • 4Account Service emits AccountDebitFailedEvent with reasonFAIL
  • 5Saga receives failure event, triggers compensation immediatelySAGA
  • 6Saga sends CancelPaymentCommand with reason INSUFFICIENT_FUNDSCOMP
  • 7Aggregate emits PaymentCancelledEvent; projection updated to CANCELLEDCANCEL
  • 8Notification Service fires failure alert. Client polls → CANCELLEDNOTIF
07 — Interview-Ready Pattern Explanations
Know These Cold
Each pattern solves a specific failure mode. When asked in an interview, explain: (1) the problem, (2) why simpler approaches fail, (3) how this pattern fixes it, (4) a concrete example from your project. 2 minutes max per pattern.
Event Sourcing

Problem: CRUD overwrites state. You can't reconstruct what happened. Solution: Store immutable events, not current state. Replay to rebuild. Your example: "PaymentAggregate state is reconstructed by replaying PaymentInitiatedEvent, then AccountDebitedEvent, then PaymentConfirmedEvent. If we need to debug why a payment failed, we have the complete event chain."

Outbox Pattern

Problem: DB write + Kafka publish = dual write. If crash between them, data diverges. Solution: Write DB + outbox in one ACID txn. Relay polls outbox → publishes to Kafka → marks published after ACK. Your example: "In my Mastercard project, we solved dual-write by using this pattern — the relay guarantees at-least-once delivery."

Saga (Orchestration)

Problem: Payment spans risk + account + notification. No distributed transaction. If account fails after risk approves, need rollback. Solution: Axon Saga tracks state. On failure → issues CancelPaymentCommand. Your example: "Saga watches for both RiskApprovedEvent and AccountDebitedEvent. Only when both arrive does it confirm. If either fails, it compensates."

Idempotent Consumers

Problem: Kafka at-least-once delivery means redelivery on crash. Double debit = real money lost. Solution: processed_events table. Check event ID before processing. Discard duplicates silently. Your example: "In Account Service, every event ID is stored. If Kafka redelivers after a consumer restart, the idempotency check prevents double-debit."

CQRS

Problem: Write model (transactional integrity) vs read model (fast queries) have different requirements. Sharing a model = neither is optimized. Solution: Commands go to Axon Aggregate. Queries hit denormalized PostgreSQL projections. Your example: "Query Service has zero joins — payment_summary is fully denormalized. Write throughput and read latency are independently scalable."

Dead Letter Topic

Problem: Poison pill messages (malformed, unprocessable) block the consumer partition forever. Solution: After N retries → route to {topic}.DLT. Inspect, fix, replay. Your example: "Notification Service uses Spring Kafka's DefaultErrorHandler with DeadLetterPublishingRecoverer. 3 retries with backoff, then DLT."

08 — Kafka Configuration Reference
Key Config Values — Know Why Each Exists
Every config value has a reason. If an interviewer asks "why is acks set to all?", you should be able to explain the tradeoff without hesitation.
Config KeyValueWhy — Explain This In Interviews
enable.idempotencetrueProducer won't duplicate messages on retry. Kafka assigns sequence numbers per partition. Required for exactly-once semantics.
transactional.idpayment-producer-1Enables Kafka transactions. Epoch fencing prevents zombie producers after restart. Set a unique ID per producer instance.
acksallLeader + all ISR replicas must ACK before producer call returns. No data loss on leader failure. Tradeoff: slightly higher latency.
isolation.levelread_committedConsumer skips uncommitted transactional messages. Prevents reading partial saga state from an in-progress transaction.
auto.offset.resetearliestNew consumer group starts from beginning. Safe to replay all events to rebuild projections from scratch.
enable.auto.commitfalseManual offset commit AFTER idempotency check + business logic. Prevents marking-consumed before processing is done.
max.in.flight.per.connection5Allows pipeline parallelism while idempotent producer preserves ordering within partition. Above 5 breaks ordering guarantee.
Outbox poll interval200msBalance between latency (events published quickly) and DB load (not hammering PostgreSQL). Configurable per environment.
